지금까지 우리는 14개의 챕터를 거치며 숨 가쁘게 달려왔습니다. **Vertex AI** 환경 설정부터 시작해, RAG 아키텍처를 설계하고, 도커 컨테이너에 담아 클라우드에 배포하는 과정까지 모든 개발 단계를 마쳤습니다. 이제 여러분의 서비스는 인터넷 세상에 공개되어 누구나 접속할 수 있는 상태가 되었습니다.

하지만 "배포했다"는 것이 곧 "완성했다"는 의미는 아닙니다. 진정한 개발자는 배포 버튼을 누른 직후부터 긴장해야 합니다. "서버가 갑자기 죽지는 않을까?", "사용자가 몰리면 비용이 얼마나 나올까?", "에러가 났을 때 원인을 어떻게 찾지?"와 같은 운영 이슈들이 기다리고 있기 때문입니다.

이번 마지막 챕터에서는 배포된 서비스가 건강하게 작동하는지 확인하는 **헬스 체크(Health Check)** 방법부터, 문제가 발생했을 때 빠르게 대처하기 위한 **로깅(Logging)**과 **모니터링**, 그리고 프로젝트를 성공적으로 마무리하기 위한 발표 및 향후 로드맵에 대해 알아보겠습니다.

## 15-1 배포 후 기본 동작 점검 체크리스트

서비스를 배포한 직후 가장 먼저 해야 할 일은 서비스가 '살아있는지' 확인하는 것입니다. 이를 전문 용어로 **헬스 체크(Health Check)**라고 합니다. 단순히 웹페이지가 뜨는지를 넘어, 내부 시스템이 정상적으로 응답할 준비가 되었는지를 기계적으로 확인하는 절차입니다.

### 15-1-1 헬스 체크 엔드포인트 구현

**Cloud Run**과 같은 오케스트레이션 도구는 컨테이너가 잘 떠 있는지 주기적으로 찔러봅니다(Probe). 이때 컨테이너가 "나 괜찮아(200 OK)"라고 응답하지 않으면, 구글 클라우드는 해당 컨테이너를 고장 난 것으로 간주하고 강제로 재시작시킵니다.

따라서 우리는 **FastAPI** 서버에 아주 가벼운 헬스 체크용 주소를 하나 만들어 두어야 합니다.



```python
@app.get("/health")
async def health_check():
    """
    로드밸런서나 Cloud Run이 서버 상태를 확인하기 위해 호출하는 API
    """
    return {"status": "ok", "service": "vertex-rag-api"}
```

이 코드는 아무런 로직 없이 그저 성공 신호만 보냅니다. 배포 후 브라우저에서 서비스 URL/health에 접속했을 때 `{"status": "ok"}`가 뜬다면, 서버 통신에는 문제가 없다는 뜻입니다.

### 15-1-2 콜드 스타트(Cold Start) 이해와 점검

**Cloud Run**의 특징 중 하나는 사용자가 없으면 서버가 꺼진다는 점(Scale to Zero)입니다. 서버가 꺼진 상태에서 첫 번째 요청이 들어오면, 서버를 다시 켜느라 평소보다 응답 속도가 느려지는데 이를 '콜드 스타트'라고 합니다.

배포 직후 첫 요청을 보내보세요. 아마 5~10초 정도 걸릴 수 있습니다. 그 이후 두 번째 요청부터는 1초 이내로 빨라질 것입니다. 운영 단계에서는 이 콜드 스타트 시간을 줄이기 위해 '최소 인스턴스(Min Instance)'를 1개 이상으로 설정하여 서버가 항상 깨어있도록 비용을 투자하는 전략을 취하기도 합니다.

> 📷 **[이미지]** Cloud Run 대시보드의 '지연 시간(Latency)' 그래프. 서버가 처음 켜질 때 그래프가 높게 치솟았다가(콜드 스타트), 이후 안정적으로 낮게 유지되는 패턴을 시각적으로 보여줌

## 15-2 에러 상황을 위한 최소한의 로그·예외 처리 포인트

"개발자는 코드를 믿지 말고, 로그(Log)를 믿어야 한다"는 말이 있습니다. 특히 클라우드 환경에서는 서버에 직접 접속해서 디버깅할 수 없기 때문에, 남겨진 로그만이 유일한 단서가 됩니다.

### 15-2-1 구조화된 로그(Structured Logging)의 중요성

초보 개발자들은 흔히 `print("에러 발생!")`과 같이 단순 텍스트로 로그를 남깁니다. 하지만 **Google Cloud Logging** 시스템에서 이를 효과적으로 검색하려면 JSON 형태로 남기는 것이 좋습니다.



**나쁜 예:**

```python
print(f"Error: {e}") # 단순 텍스트라 검색이 어려움
```

**좋은 예 (JSON):**

```python
import logging

# ... 설정 생략 ...

logger.error({
    "event": "rag_generation_failed",
    "error_message": str(e),
    "user_input": request.question,
    "severity": "ERROR"
})
```

이렇게 JSON으로 로그를 남기면, 나중에 "event가 'rag_generation_failed'인 로그만 찾아줘"라고 필터링하여 문제의 원인을 순식간에 파악할 수 있습니다.

### 15-2-2 필수 예외 처리 포인트

RAG 시스템에서 반드시 try-except로 감싸야 하는 위험 구간들이 있습니다.

- **Vertex AI API 호출 구간:** 구글 서버가 일시적으로 응답하지 않거나, 할당량(Quota)을 초과했을 때를 대비해야 합니다.
- **벡터 DB 검색 구간:** DB 연결이 끊기거나 검색 결과가 0개일 때, 프로그램이 멈추지 않고 "관련 문서를 찾을 수 없습니다"라고 우아하게 대처해야 합니다.

> 📷 **[이미지]** Google Cloud Logging 콘솔 화면. 수많은 로그 중에서 특정 JSON 필드(예: severity="ERROR")를 클릭하여 필터링하는 모습

## 15-3 Monitoring으로 비용과 성능 감시하기

서비스가 안정적으로 돌아간다면, 다음 관심사는 '돈'과 '성능'입니다. **Vertex AI**와 **Cloud Run**은 사용한 만큼 비용을 내기 때문에, 방심하면 요금 폭탄을 맞을 수 있습니다.

### 15-3-1 비용 관리 및 알림 설정 (Billing Alert)

우리는 앞선 챕터에서 예산 알림을 설정했습니다. 프로젝트 마무리 시점에는 실제 비용이 얼마나 발생했는지 '결제 보고서'를 확인해 보는 습관을 들여야 합니다.

**주요 비용 발생원:**

- **Gemini API**의 토큰 사용량(입력/출력 글자 수)
- **Cloud Run**의 CPU 가동 시간

특히 RAG 시스템은 문서를 검색해서 프롬프트에 넣기 때문에 입력 토큰(Input Token) 비용이 많이 발생할 수 있음을 인지해야 합니다.

### 15-3-2 지연 시간(Latency) 모니터링

생성형 AI 서비스의 품질은 '속도'에 달려 있습니다. 사용자는 답변이 3초 이상 늦어지면 답답함을 느낍니다. **Cloud Monitoring** 대시보드에서 다음 두 가지 지표를 주시해야 합니다.

- **P50 Latency:** 평균적인 응답 속도.
- **P99 Latency:** 가장 느린 상위 1%의 응답 속도. (가장 불만족한 사용자의 경험)

만약 P99가 너무 높다면, 벡터 DB 검색 속도가 느리거나 프롬프트가 지나치게 길지 않은지 의심해봐야 합니다.

> 📷 **[이미지]** 구글 클라우드 결제(Billing) 리포트 화면. 날짜별 비용 막대그래프와 함께, 우측에 서비스별(Vertex AI, Cloud Run) 비용 비중이 파이 차트로 나타난 이미지
