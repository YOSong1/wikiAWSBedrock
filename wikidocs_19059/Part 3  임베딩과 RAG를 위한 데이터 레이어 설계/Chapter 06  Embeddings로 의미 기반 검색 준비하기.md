## 6-1 임베딩(Embeddings)의 개념과 활용 사례

우리는 지금까지 Gemini라는 똑똑한 AI 모델과 대화하는 방법을 배웠습니다. 하지만 Gemini는 세상의 모든 지식을 알고 있을지 몰라도, 우리 회사의 비공개 문서나 방금 작성된 내부 보고서는 알지 못합니다. AI에게 이 지식을 전달하려면 어떻게 해야 할까요?

단순히 문서를 텍스트로 복사해서 프롬프트에 붙여넣는 것은 한계가 있습니다. 문서의 양이 방대하기 때문입니다. 그래서 우리는 방대한 데이터 속에서 "질문과 관련된 부분만 쏙 뽑아내는 기술"이 필요합니다. 그 핵심 기술이 바로 **임베딩(Embeddings)**입니다.

이번 챕터에서는 컴퓨터가 사람의 언어를 이해하는 방식인 **임베딩**의 개념을 익히고, **Vertex AI**를 통해 텍스트를 '의미를 가진 숫자 벡터'로 변환하는 실습을 진행합니다.

> **💡 핵심 포인트**
> 임베딩은 단순 키워드 검색의 한계를 극복하고, 텍스트의 의미를 숫자로 표현하여 의미 기반 검색을 가능하게 하는 핵심 기술입니다.

### 6-1-1 키워드 검색 vs 의미 기반 검색

우리가 흔히 쓰는 '검색(Ctrl+F)'은 글자가 일치하는지를 봅니다. 예를 들어 "맛있는 과일"을 검색하면, 정확히 그 단어가 들어간 문서는 찾지만 "달콤한 사과"가 적힌 문서는 찾지 못합니다. 글자가 다르기 때문입니다.

하지만 임베딩은 다릅니다. 임베딩은 텍스트를 수백, 수천 개의 숫자로 이루어진 리스트(벡터)로 변환합니다. 이 숫자들은 단어의 **'의미'**를 나타내는 좌표입니다.

- **키워드 검색**: "강아지" ≠ "멍멍이" (다른 글자로 인식)
- **임베딩(의미) 검색**: "강아지" ≒ "멍멍이" (의미 공간 상에서 매우 가까운 위치에 존재)

### 6-1-2 벡터 공간에서의 의미 거리

**임베딩**을 통해 변환된 숫자들을 2차원 공간에 점으로 찍었다고 상상해 봅시다.

- '왕(King)'과 '남자(Man)'라는 점 사이의 거리는 가깝습니다.
- '여왕(Queen)'과 '여자(Woman)' 사이의 거리도 가깝습니다.

놀랍게도, (왕 - 남자 + 여자) 연산을 하면 **'여왕'**이라는 위치에 도달합니다.

이처럼 **임베딩**은 컴퓨터가 단어의 의미와 관계를 수학적으로 계산할 수 있게 해줍니다.

> 📷 **[이미지]** 2차원 또는 3차원 공간에 단어들이 점으로 찍혀 있는 그림. 비슷한 의미(사과, 바나나)끼리 뭉쳐 있고, 다른 의미(자동차, 비행기)는 멀리 떨어져 있는 '벡터 공간' 시각화

### 6-1-3 주요 활용 사례

1. **의미 기반 검색 (Semantic Search)**: 사용자가 "배송이 왜 이렇게 늦어?"라고 화난 말투로 검색해도, "배송 지연 보상 정책" 문서를 찾아줄 수 있습니다.

2. **추천 시스템**: 내가 본 뉴스 기사와 '의미적으로 거리가 가까운' 기사를 추천합니다.

3. **분류 (Classification)**: 고객의 리뷰 텍스트를 **임베딩**하여 긍정/부정 벡터와 비교해 감정을 분석합니다.

## 6-2 문서/단락/문장 단위 임베딩 비교

텍스트를 벡터로 변환할 때, "얼마나 많은 양의 텍스트를 하나의 벡터로 만들 것인가?"는 매우 중요한 문제입니다. 이를 **청킹(Chunking)** 단위라고도 합니다.

### 6-2-1 문서(Document) 단위

문서 전체(예: PDF 파일 하나)를 하나의 벡터로 만듭니다.

**장점**: 전체적인 주제를 파악하기 좋습니다.

**단점**: 문서 안에 다양한 주제가 섞여 있을 경우, 의미가 **희석(Dilution)**되어 검색 정확도가 떨어집니다. "아이폰 수리 매뉴얼" 전체를 **임베딩**하면, 구체적인 "배터리 교체 방법"을 질문했을 때 찾기 어려울 수 있습니다.

### 6-2-2 문장(Sentence) 단위

마침표를 기준으로 문장 하나하나를 벡터로 만듭니다.

**장점**: 아주 구체적인 사실을 찾을 때 정확합니다.

**단점**: 문맥이 잘려나갈 수 있습니다. "그것은 매우 위험합니다."라는 문장만으로는 '그것'이 무엇인지 알 수 없게 됩니다.

### 6-2-3 단락(Paragraph) 또는 고정 크기 단위

보통 300~500자 정도의 의미 있는 단락 단위로 자릅니다.

**특징**: **RAG** 시스템에서 가장 널리 사용되는 방식입니다. 적절한 문맥을 포함하면서도, 구체적인 내용을 검색할 수 있는 균형 잡힌 단위입니다.

우리는 앞으로 진행할 프로젝트에서, 문서를 적절한 크기의 **단락(Chunk)**으로 나누어 **임베딩**을 수행할 것입니다.

> 📷 **[이미지]** 하나의 긴 문서를 통째로 임베딩했을 때의 벡터(하나의 점)와, 문단별로 쪼개서 임베딩했을 때의 벡터들(여러 개의 점)을 비교하는 다이어그램

## 6-3 Vertex AI Embeddings API 호출 및 실습

이제 이론을 바탕으로 실제 텍스트를 숫자로 바꿔보겠습니다. **Vertex AI**는 `text-embedding-004`와 같은 고성능 **임베딩 모델**을 API로 제공합니다. 특히 한국어를 포함한 **다국어(Multilingual)** 지원이 뛰어납니다.

### 6-3-1 텍스트 임베딩 모델 로드

VS Code에서 새로운 노트북 파일(`embedding_test.ipynb`)을 만들고 아래 코드를 작성합니다.

```python
from vertexai.language_models import TextEmbeddingModel

# 1. 임베딩 모델 로드 (최신 다국어 모델 사용)
embedding_model = TextEmbeddingModel.from_pretrained("text-multilingual-embedding-002")

def get_embedding(text):
    # 텍스트를 입력받아 벡터(숫자 리스트)를 반환하는 함수
    embeddings = embedding_model.get_embeddings([text])
    return embeddings[0].values

# 테스트
text = "Vertex AI는 Google Cloud의 머신러닝 플랫폼입니다."
vector = get_embedding(text)

# 결과 확인
print(f"텍스트: {text}")
print(f"벡터 길이(차원 수): {len(vector)}")
print(f"벡터 앞부분 10개: {vector[:10]}")
```

### 6-3-2 실행 결과 해석

코드를 실행하면 `[0.012, -0.045, 0.033, ...]`과 같은 실수(float) 리스트가 출력됩니다. 보통 **768차원**(숫자 768개)의 리스트가 생성됩니다. 이 숫자들의 조합이 바로 "Vertex AI는 머신러닝 플랫폼이다"라는 문장의 고유한 지문이 되는 것입니다.

> **📌 참고:**
> 각 차원의 숫자들은 텍스트의 의미 정보를 다차원 공간에 인코딩한 형태입니다. 이 벡터들을 비교하면 의미적 유사성을 계산할 수 있습니다.

### 6-3-3 유사도 비교 실습

임베딩의 위력을 확인하기 위해 간단한 비교 실험을 해봅시다.

```python
import numpy as np

# 두 벡터 사이의 유사도를 계산하는 함수 (코사인 유사도)
def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

vec1 = get_embedding("맛있는 사과")
vec2 = get_embedding("달콤한 과일")
vec3 = get_embedding("빠른 자동차")

score1 = cosine_similarity(vec1, vec2)  # 사과 vs 과일
score2 = cosine_similarity(vec1, vec3)  # 사과 vs 자동차

print(f"사과 vs 과일 유사도: {score1:.4f}")
print(f"사과 vs 자동차 유사도: {score2:.4f}")
```

실행해 보면 '사과 vs 과일'의 점수가 '사과 vs 자동차'보다 월등히 높게 나올 것입니다. 컴퓨터가 단어의 의미를 이해하고 있다는 증거입니다.

> 📷 **[이미지]** 위 코드의 실행 결과 화면. 벡터 리스트가 출력된 모습과, 유사도 점수(예: 0.85 vs 0.12)가 비교된 터미널 결과

## 6-4 임베딩 결과 저장 형식(벡터 + 메타데이터) 설계

**임베딩** API를 통해 얻은 것은 단순히 숫자 덩어리(벡터)뿐입니다. 나중에 검색 결과를 사용자에게 보여주려면, 이 숫자가 원래 어떤 문장의 내용이었는지, 출처는 어디인지를 함께 저장해야 합니다.

### 6-4-1 벡터와 메타데이터의 결합

데이터를 저장할 때는 **벡터(Vector)**와 **메타데이터(Metadata/Payload)**를 한 쌍으로 묶어서 관리해야 합니다.

- **벡터**: 검색(계산) 용도 → `[0.1, 0.5, ...]`
- **메타데이터**: 결과 표시 용도
  - `content`: "Vertex AI는 Google Cloud의..." (원본 텍스트)
  - `source`: "vertex_guide.pdf" (파일 출처)
  - `page`: 15 (페이지 번호)

### 6-4-2 JSON 구조

실제 데이터베이스에 저장될 데이터의 형태는 다음과 같은 JSON 구조를 가집니다.

```json
{
  "id": "doc_001_chunk_05",
  "values": [0.0123, -0.0456, ...],
  "metadata": {
    "text": "임베딩은 텍스트를 벡터로 변환하는 기술입니다.",
    "filename": "lecture_note_ch6.pdf",
    "page_number": 3,
    "created_at": "2025-05-20"
  }
}
```

이 구조를 잘 설계해야 나중에 "2024년도 문서 중에서만 검색해줘"와 같은 **필터링(Filtering)** 기능을 구현할 수 있습니다.

> 📷 **[이미지]** 텍스트 문서가 임베딩 모델을 통과하여 벡터가 되고, 여기에 메타데이터가 붙어서 최종적으로 데이터베이스 저장용 패키지가 되는 흐름도 (Input Text → Model → Vector + Metadata → DB)
