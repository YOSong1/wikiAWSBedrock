지난 Chapter 6에서 우리는 텍스트를 AI가 이해할 수 있는 숫자(벡터)로 바꾸는 **임베딩** 기술을 익혔습니다. 하지만 현실 세계의 데이터는 우리가 실습했던 깔끔한 한 문장짜리 텍스트가 아닙니다. 회사 내의 데이터는 복잡한 표와 서식이 섞인 PDF 파일, 수십 페이지에 달하는 보고서, 혹은 엉성하게 작성된 메모 형태로 존재합니다.

이런 거칠고 방대한 원본 데이터를 그대로 **임베딩 모델**에 넣을 수는 없습니다. 마치 요리를 하기 전에 재료를 씻고 다듬는 과정이 필요하듯, AI에게 데이터를 먹이기 전에도 **전처리(Preprocessing)**와 **청킹(Chunking)**이라는 과정이 반드시 필요합니다.

이번 챕터에서는 **RAG** 시스템의 성능을 좌우하는 데이터 파이프라인을 구축해보겠습니다. PDF 문서를 텍스트로 추출하고, 이를 AI가 읽기 좋은 크기로 자르는 전략을 상세히 다룹니다.

> **💡 핵심 포인트:**
> 원본 데이터의 품질과 청킹 방식이 RAG 시스템의 검색 정확도를 결정하는 가장 중요한 요소입니다.

## 7-1 PDF/텍스트 문서 파싱 전략

데이터 파이프라인의 첫 단계는 **파싱(Parsing)**입니다. 파싱이란 사람이 보는 문서 파일(PDF, PPTX, DOCX 등)에서 컴퓨터가 처리할 수 있는 순수한 텍스트 데이터만을 뽑아내는 과정을 말합니다.

### 7-1-1 PDF 파싱의 어려움

특히 비즈니스 환경에서 가장 많이 쓰이는 **PDF** 파일은 AI에게 가장 까다로운 상대입니다. PDF는 텍스트의 구조(문단, 제목)보다는 '보여지는 위치' 정보를 중심으로 저장되기 때문입니다. 단순히 텍스트를 긁어오면 단의 순서가 뒤섞이거나, 머리말/꼬리말이 본문 중간에 끼어들어가 문맥을 망치기도 합니다.

> **⚠️ 주의:**
> PDF 파일에서 텍스트를 추출할 때는 레이아웃과 구조 정보가 손실될 수 있으므로 주의가 필요합니다.

### 7-1-2 도구 선택: LangChain Document Loaders

이러한 복잡함을 해결하기 위해 우리는 **LangChain**이라는 프레임워크를 활용할 것입니다. **LangChain**은 다양한 포맷의 문서를 손쉽게 불러올 수 있는 **Document Loader** 기능을 제공합니다.

- **PyPDFLoader**: 가장 기본적이고 빠릅니다. 텍스트 위주의 간단한 PDF에 적합합니다.
- **PDFPlumber**: 표나 복잡한 레이아웃 처리에 좀 더 강력합니다.
- **UnstructuredLoader**: 이미지, 표, 텍스트가 섞인 복잡한 문서에 강력하지만 설치가 조금 더 까다롭습니다.

이번 실습에서는 가장 대중적인 **PyPDFLoader**를 사용하여 문서를 로드해 보겠습니다.

> 📷 **[이미지]** 다양한 문서 아이콘(PDF, Word, HTML)이 깔때기(Loader)를 통과하여 균일한 텍스트 데이터로 변환되어 나오는 개념도. 파싱의 역할을 시각적으로 보여주는 그림

## 7-2 청크 크기, 중첩, 태그(메타데이터) 설계 기준

문서에서 텍스트를 추출했다면, 이제 이를 적절한 크기로 잘라야 합니다. 이 과정을 **청킹(Chunking)**이라고 합니다. **청킹**은 **RAG** 시스템의 검색 정확도를 결정하는 매우 중요한 변수입니다.

### 7-2-1 청킹의 필요성

- **입력 길이 제한**: **임베딩 모델**과 **LLM**은 한 번에 처리할 수 있는 단어 수(토큰)에 한계가 있습니다. 책 한 권을 통째로 넣을 수 없습니다.

- **검색의 정확도**: 문서 전체를 하나로 묶으면 주제가 희석됩니다. 사용자가 "배터리 교체 방법"을 물었을 때, 문서 전체보다는 해당 내용이 적힌 '한 문단'을 찾아주는 것이 훨씬 정확합니다.

### 7-2-2 청크 크기 (Chunk Size)

청크 크기는 한 번에 자를 글자(또는 토큰)의 수를 의미합니다.

- **너무 작으면** (예: 50자): 문맥이 잘려 의미 파악이 안 됩니다. "그것을 눌러라"라는 문장에서 '그것'이 무엇인지 알 수 없게 됩니다.

- **너무 크면** (예: 2000자): 불필요한 정보(Noise)가 많이 섞여 검색 정확도가 떨어집니다.

- **권장 크기**: 일반적으로 500~1000자(token) 정도가 문맥을 유지하면서 검색하기에 가장 적절한 크기로 알려져 있습니다.

### 7-2-3 중첩 (Chunk Overlap) - "슬라이딩 윈도우"

문서를 단순히 칼로 무 자르듯 뚝뚝 끊으면 문제가 발생합니다. 하필 중요한 문장이 잘리는 경계선에 위치한다면 어떻게 될까요? 의미가 반으로 나뉘어 검색되지 않을 것입니다.

이를 방지하기 위해 **중첩(Overlap)**을 둡니다. 앞 **청크**의 끝부분을 뒷 **청크**의 앞부분에 10~20% 정도 겹치게 포함시키는 것입니다. 이를 통해 문맥의 끊김 없이 부드럽게 연결할 수 있습니다.

> **📌 참고:**
> 중첩을 사용하면 데이터 크기가 증가하지만, 검색 품질과 문맥 보존 측면에서 큰 이점이 있습니다.

### 7-2-4 메타데이터 (Metadata) 설계

잘라낸 텍스트 조각에는 반드시 '꼬리표'를 붙여야 합니다.

- **source**: 파일명 (예: "2025_HR_Guide.pdf")
- **page**: 페이지 번호 (예: 15쪽)
- **category**: 문서 분류 (예: "인사규정")

이 **메타데이터**가 있어야 나중에 **LLM**이 답변할 때 "이 내용은 인사규정 문서 15쪽에서 참고했습니다"라고 근거를 제시할 수 있습니다.

> 📷 **[이미지]** 긴 텍스트 띠를 일정한 간격(Chunk Size)으로 자르되, 연결 부위가 겹치도록(Overlap) 겹쳐서 자르는 '슬라이딩 윈도우' 방식의 다이어그램 ([Chunk 1 ... overlap ... Chunk 2] 형태)

## 7-3 문서 청크 임베딩 생성 파이프라인 구현

이제 이론을 바탕으로 실제 코드를 작성해 보겠습니다. 로컬 폴더에 있는 **PDF** 파일을 읽어와서, 적절한 크기로 자르고(Split), 각 조각의 정보를 확인하는 파이프라인을 구축합니다.

### 7-3-1 환경 설정 및 라이브러리 임포트

먼저 필요한 LangChain 라이브러리를 설치하고 임포트합니다.

```bash
pip install langchain langchain-community pypdf
```

```python
from langchain_community.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter

# 실습할 PDF 파일 경로 지정 (프로젝트 폴더에 pdf 파일을 미리 넣어두세요)
FILE_PATH = "./sample_data/vertex_guide.pdf"
```

### 7-3-2 문서 로드 및 텍스트 분할기 설정

가장 널리 쓰이는 **RecursiveCharacterTextSplitter**를 사용합니다. 이 분할기는 문단(`\n\n`) → 문장(`\n`) → 단어(공백) 순서로 최대한 의미 단위가 깨지지 않도록 똑똑하게 자릅니다.

```python
# 1. 문서 로드
loader = PyPDFLoader(FILE_PATH)
documents = loader.load()
print(f"총 페이지 수: {len(documents)}")

# 2. 분할기(Splitter) 설정
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,  # 청크 당 약 500자
    chunk_overlap=50,  # 50자 정도 겹치게 설정
    separators=["\n\n", "\n", " ", ""]  # 자르는 우선순위
)

# 3. 문서 분할 실행
split_docs = text_splitter.split_documents(documents)
print(f"분할된 총 청크 수: {len(split_docs)}")
```

### 7-3-3 분할 결과 확인

잘라진 데이터가 어떻게 생겼는지 눈으로 직접 확인해야 합니다.

```python
# 첫 번째 청크의 내용과 메타데이터 확인
first_chunk = split_docs[0]
print("=== 첫 번째 청크 내용 ===")
print(first_chunk.page_content)
print("\n=== 메타데이터 ===")
print(first_chunk.metadata)
```

이 코드를 실행하면 PDF 내용이 500자 단위로 잘려 있고, metadata에 `{'source': ..., 'page': ...}` 정보가 자동으로 들어가 있는 것을 확인할 수 있습니다. 이제 이 `split_docs` 리스트를 그대로 지난 챕터에서 배운 **임베딩 모델**에 넣기만 하면 됩니다.

> 📷 **[이미지]** VS Code 터미널에서 위 코드를 실행한 결과 화면. "총 페이지 수: 10", "분할된 총 청크 수: 35" 등의 로그와 함께, 실제 텍스트 내용과 메타데이터 딕셔너리가 출력된 모습

## 7-4 품질 좋은 RAG를 위한 데이터 구성 팁

마지막으로, 단순히 자르는 것을 넘어 현업에서 검색 품질을 높이기 위해 사용하는 몇 가지 '노하우'를 공유합니다.

### 7-4-1 불필요한 요소 제거 (Cleaning)

PDF의 매 페이지마다 반복되는 "2025년 사업보고서 - 대외비" 같은 머리말이나 바닥글은 제거하는 것이 좋습니다. 이런 문구가 모든 **청크**에 들어가면, 검색 모델이 이 반복 문구에 과도하게 주목하여 검색 품질을 떨어뜨릴 수 있습니다. **정규표현식(Regex)** 등을 활용해 패턴화된 노이즈를 지워주세요.

> **⚠️ 주의:**
> 노이즈가 많으면 임베딩 모델이 의미 있는 정보보다 반복되는 불필요한 패턴에 더 가중치를 두게 됩니다.

### 7-4-2 표(Table) 데이터 처리의 중요성

**RAG** 시스템의 가장 큰 난적은 '표'입니다. 표를 단순히 줄글로 풀면 행과 열의 구조가 깨져서 "매출액"이 "1분기"인지 "2분기"인지 알 수 없게 됩니다.

> **💡 팁:**
> 표는 별도로 추출하여 **마크다운(Markdown)** 형식으로 변환하거나, **LLM**에게 "이 표를 요약해줘"라고 시켜서 그 요약본을 **임베딩**하는 것이 검색에 훨씬 유리합니다.

### 7-4-3 제목과 본문의 결합

**청크**를 자르다 보면 소제목은 앞 **청크**에, 본문은 뒤 **청크**에 떨어지는 경우가 있습니다. 이를 방지하기 위해 **MarkdownHeaderTextSplitter** 같은 도구를 사용하면, `[챕터1 > 섹션2 > 본문]`과 같이 계층 구조를 유지하면서 자를 수 있어 검색 맥락을 훨씬 잘 보존할 수 있습니다.

> 📷 **[이미지]** '나쁜 청킹' 예시(표가 깨져서 의미를 알 수 없는 텍스트)와 '좋은 청킹' 예시(마크다운으로 표 구조가 유지되거나 요약된 텍스트)를 비교하는 이미지
