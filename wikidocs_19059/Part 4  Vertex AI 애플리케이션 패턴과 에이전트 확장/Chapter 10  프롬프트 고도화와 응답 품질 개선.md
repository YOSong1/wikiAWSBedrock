앞선 챕터에서 우리는 검색된 정보를 바탕으로 답변하는 **RAG 시스템**을 구축했습니다. 하지만 "기능이 동작한다"는 것과 "사용자가 만족한다"는 것은 전혀 다른 차원의 문제입니다. 때로는 AI가 엉뚱한 동문서답을 하기도 하고, 비즈니스 매너에 맞지 않는 가벼운 말투를 쓰기도 하며, 심지어는 없던 사실을 지어내기도 합니다.

이러한 문제들을 해결하기 위해 모델 자체를 새로 학습시키는 것은 비용과 시간이 많이 듭니다. 대신 우리는 **프롬프트(Prompt)**, 즉 AI에게 지시를 내리는 방식만 정교하게 다듬어도 놀라운 성능 향상을 이뤄낼 수 있습니다. 이를 **프롬프트 엔지니어링(Prompt Engineering)**이라고 합니다.

이번 챕터에서는 단순한 지시를 넘어, 예시를 통해 학습시키는 **Few-shot** 기법, 논리적 사고를 유도하는 **Chain-of-Thought(CoT)**, 그리고 가장 치명적인 문제인 **환각(Hallucination)**을 제어하는 고급 프롬프트 설계 전략을 상세히 다루겠습니다.

> **💡 핵심 포인트:** 프롬프트 엔지니어링은 모델 재학습 없이 응답 품질을 획기적으로 개선할 수 있는 강력한 기법입니다.

## 10-1 Few-shot, Chain-of-Thought, 단계적 추론 적용

**Gemini**와 같은 **거대 언어 모델(LLM)**은 기본적으로 매우 똑똑하지만, 때로는 구체적인 가이드라인이 없으면 엉뚱한 방향으로 답변을 생성합니다. 이를 보정하기 위한 가장 강력한 두 가지 무기가 바로 **Few-shot(퓨샷)**과 **Chain-of-Thought(사고의 사슬)**입니다.

### 10-1-1 Zero-shot vs Few-shot

우리가 지금까지 사용한 방식은 예시 없이 질문만 던지는 **Zero-shot** 방식이었습니다. 하지만 신입 사원에게 업무를 지시할 때를 생각해 보십시오. "보고서 써와"라고 말하는 것보다, "지난달에 김 대리가 쓴 이 보고서를 참고해서 비슷하게 써와"라고 예시를 주는 것이 훨씬 효과적입니다.

**Few-shot 프롬프팅**은 프롬프트 안에 **질문**과 **이상적인 답변**의 예시를 몇 가지(Few) 포함시켜서 모델에게 전달하는 기법입니다.

**Zero-shot:**

```
"이 리뷰의 감정을 분석해줘: '음식이 식어서 왔어요.'"
```

**Few-shot:**

```
다음은 고객 리뷰에 대한 감정 분석 예시입니다.

리뷰: "배송이 정말 빠르네요!"
감정: 긍정

리뷰: "박스가 찌그러져서 왔어요."
감정: 부정

리뷰: "음식이 식어서 왔어요."
감정:
```

이렇게 예시를 주면 모델은 "아, 이런 형식으로 짧게 답하라는 거구나"라고 패턴을 즉각적으로 파악하여, "부정"이라고 정확하게 답변합니다. Vertex AI SDK에서는 `examples` 파라미터를 통해 이를 구조적으로 지원합니다.

> 📷 **[이미지]** Zero-shot 프롬프팅과 Few-shot 프롬프팅의 구조 비교. Zero-shot은 [지시문]->[질문]으로 바로 이어지지만, Few-shot은 [지시문]->[예시1]->[예시2]->[질문]으로 중간 단계가 있음을 시각화.

### 10-1-2 Chain-of-Thought (CoT): 단계적 추론

복잡한 수학 문제나 추론 문제를 풀 때, 단순히 정답만 요구하면 AI도 틀릴 확률이 높습니다. 하지만 "단계적으로 생각해서 풀이 과정을 적어줘"라고 지시하면 정답률이 비약적으로 상승합니다. 이것이 바로 **Chain-of-Thought(CoT)** 기법입니다.

사람도 암산보다 연습장에 풀이를 적을 때 실수가 적은 것처럼, 모델에게도 답변을 생성하기 전에 **생각할 시간(토큰)**을 부여하는 원리입니다.

- **일반 프롬프트**: "사과 5개가 있는데 2개를 먹고 3개를 더 샀어. 몇 개야?"
- **CoT 프롬프트**: "사과 개수 문제를 해결해줘. 단계별로 차근차근 생각해보자 (Let's think step by step)."

프롬프트 끝에 마법의 문장 **"Let's think step by step"**을 붙이는 것만으로도 모델은 내부적으로 추론 과정을 거치게 되어 논리적인 오류를 스스로 수정합니다.

RAG 시스템에서도 "먼저 검색된 문서를 분석하고, 그 다음에 답변을 작성해"라고 단계를 나누어 지시하면 훨씬 더 정확한 결과를 얻을 수 있습니다.

> 📷 **[이미지]** CoT가 적용되었을 때 모델 내부의 처리 과정. [입력] -> [중간 추론 단계 생성: "5-2=3, 3+3=6"] -> [최종 답변: 6]으로 이어지는 흐름을 도식화.

## 10-2 도메인 전용 지시문과 스타일 프리셋 설계

RAG 시스템이 범용적인 챗봇이 아니라, 특정 비즈니스 도메인(법률, 의료, 금융 등)의 전문가처럼 행동하게 하려면 그에 맞는 **페르소나(Persona)**와 **스타일(Style)**을 입혀야 합니다.

### 10-2-1 도메인 전용 지시문 (System Instructions)

Chapter 5에서 잠깐 다루었던 **System Instruction**을 심화하여 적용해 보겠습니다. 단순히 "친절하게 답해" 수준이 아니라, **도메인 특화 지식**을 전제 조건으로 깔아주는 것입니다.

예를 들어 '사내 IT 헬프데스크 봇'을 만든다면 다음과 같이 구체적인 시스템 지시문을 작성해야 합니다.

```
당신은 10년 차 IT 시스템 관리자입니다.
비전문가인 직원들에게 기술적인 문제를 아주 쉽게 설명해야 합니다.
답변 시 다음 원칙을 따르세요:

1. 전문 용어(예: DNS, DHCP)를 사용할 때는 반드시 괄호 안에 풀어서 설명하세요.
2. 해결 방법은 항상 1, 2, 3 번호가 매겨진 리스트 형태로 제공하세요.
3. 마지막에는 "해결되지 않으면 02-1234로 전화주세요"라는 문구를 덧붙이세요.
```

이 지시문이 적용되면, 사용자가 "인터넷이 안 돼요"라고 물었을 때 AI는 자동으로 **1, 2, 3 단계**의 해결책을 제시하고 **연락처**까지 안내하게 됩니다.

> **📌 참고:** 페르소나 설정은 사용자 경험의 일관성을 높이고 신뢰도를 강화합니다.

### 10-2-2 출력 스타일 프리셋 (Output Formatting)

답변의 내용뿐만 아니라 **형식**도 중요합니다. 개발자를 위한 봇이라면 코드는 반드시 **마크다운 블록(```)**으로 감싸야 하고, 데이터 분석가를 위한 봇이라면 결과를 **JSON**이나 **표(Table)** 형태로 출력해야 할 것입니다.

프롬프트에 **출력 템플릿**을 미리 정의해 주면 **일관된 스타일**을 유지할 수 있습니다.

```
[요청]
다음 회의록을 요약해줘.

[출력 형식]
- **주요 안건**: (한 줄 요약)
- **결정 사항**: (글머리 기호로 나열)
- **Action Item**: (담당자) - (할 일)
```

이렇게 형식을 강제하면, 사용자는 매번 똑같은 구조의 깔끔한 요약본을 받아볼 수 있어 **업무 효율**이 크게 향상됩니다.

> 📷 **[이미지]** 동일한 질문("클라우드란 무엇인가?")에 대해, '유치원 선생님 페르소나'로 답변했을 때와 'IT 엔지니어 페르소나'로 답변했을 때의 텍스트를 좌우로 비교.

## 10-3 환각(Hallucination) 완화를 위한 프롬프트 패턴

생성형 AI의 가장 큰 리스크는 사실이 아닌 것을 사실처럼 말하는 **환각(Hallucination)** 현상입니다. 특히 기업용 서비스에서 잘못된 정보 제공은 치명적일 수 있습니다. 이를 100% 막을 수는 없지만, **프롬프트 엔지니어링**을 통해 최소화할 수는 있습니다.

> **⚠️ 주의:** 환각은 모델의 학습 과정에서 발생하는 근본적인 문제로, 프롬프트 설계로만 완전히 제거할 수는 없습니다.

### 10-3-1 지식의 경계 설정: "모르면 모른다고 해라"

가장 기본적이면서도 효과적인 방법은 AI에게 **모름을 인정하는 법**을 가르치는 것입니다. AI는 기본적으로 질문에 답을 하려는 성향이 강하기 때문에, 이를 억제하는 **명시적인 지시**가 필요합니다.

```
"제공된 [Context] 내에서만 답변을 찾으세요. 만약 문맥 안에 정답이 없다면, 절
대 외부 지식을 사용하여 지어내지 말고 '죄송합니다. 제공된 문서에서 해당 정
보를 찾을 수 없습니다.'라고만 답변하세요."
```

이 문구를 프롬프트에 포함시키는 것만으로도, **엉뚱한 거짓 답변**을 획기적으로 줄일 수 있습니다.

> **📌 참고:** 지식 경계를 명확하게 설정하는 것은 환각 방지의 첫 번째 방어선입니다.

### 10-3-2 근거 기반 추론 유도 (Grounding)

모델에게 답변을 생성할 때 반드시 **문서의 어느 부분에서 가져왔는지**를 인용하도록 강제하는 방법입니다.

```
"답변의 각 문장 끝에는 반드시 참조한 문서의 페이지 번호를 [Page: 5] 형식으로
표기하세요. 참조할 부분이 없다면 문장을 생성하지 마세요."
```

이렇게 하면 모델은 근거가 확실하지 않은 문장은 스스로 생성을 포기하게 되어, 결과적으로 **신뢰할 수 있는 정보**만 남게 됩니다.

> **📌 참고:** 출처 표기 강제는 모델의 자기 검증 능력을 자극하는 효과적인 방법입니다.

### 10-3-3 사실 검증 단계 추가 (Self-Correction)

고급 기법으로, 모델에게 답변을 생성하게 한 뒤 "방금 한 답변이 문맥에 비추어 사실인지 스스로 검증해봐"라고 한 번 더 묻는 과정을 추가할 수 있습니다. **자기 검토 과정(Self-Reflection)**을 통해 미묘한 오류를 수정하게 만드는 것입니다.

> 📷 **[이미지]** 환각 현상 방지를 위한 '안전장치 프롬프트' 예시. [Context]에 없는 질문("내일 날씨 어때?")이 들어왔을 때, 나쁜 답변("내일은 맑을 것입니다.")과 좋은 답변("문서에 날씨 정보가 없어 알 수 없습니다.")을 대비시킴.

## 10-4 간단한 A/B 테스트로 프롬프트 비교 평가하기

지금까지 배운 다양한 기법들을 적용하여 프롬프트를 수정했습니다. 그런데, 수정된 프롬프트가 이전보다 더 좋다는 것을 어떻게 확신할 수 있을까요? 감에 의존하지 않고 **데이터**로 증명하기 위해 **A/B 테스트**가 필요합니다.

### 10-4-1 프롬프트 버전 관리

개발 코드처럼 프롬프트도 **버전 관리**가 필요합니다.

- **Prompt A (기본형)**: "문서를 보고 답변해줘."
- **Prompt B (고도화형)**: "IT 전문가 페르소나 + CoT 적용 + 환각 방지 문구 포함."

이 두 가지 버전을 준비해 두고, 동일한 **질문 세트(테스트 케이스)**를 던져서 결과를 비교해야 합니다.

> **📌 참고:** 버전 관리는 프롬프트 개선 과정을 추적 가능하게 만드는 핵심입니다.

### 10-4-2 평가 기준 수립 (Evaluation Metrics)

무엇이 **좋은 답변**인지 기준을 세워야 합니다.

- **정확성(Accuracy)**: 문서의 내용과 일치하는가?
- **완결성(Completeness)**: 질문에 대한 답을 빠짐없이 포함했는가?
- **친절도/톤앤매너**: 지정한 페르소나를 잘 유지했는가?

### 10-4-3 자동화된 평가 스크립트 실습

일일이 사람이 읽어보고 점수를 매기는 것은 힘든 일입니다. 놀랍게도, 평가 자체를 **LLM**에게 맡길 수 있습니다. (**LLM-as-a-Judge**)

```python
# 간단한 A/B 테스트 및 평가 의사코드
questions = ["질문1", "질문2", "질문3"]
prompt_a_results = []
prompt_b_results = []

for q in questions:
    ans_a = model.generate(prompt_a + q)
    ans_b = model.generate(prompt_b + q)

    # 평가용 LLM에게 두 답변을 비교하게 시킴
    eval_prompt = f"""
질문: {q}
답변 A: {ans_a}
답변 B: {ans_b}
어느 답변이 더 정확하고 친절한지 평가하고, 승자를 선택해줘.
"""
    verdict = evaluator_model.generate(eval_prompt)
    print(f"질문: {q} => 승자: {verdict}")
```

이러한 간단한 스크립트를 통해 우리는 "**Prompt B**가 **Prompt A**보다 80% 더 우수한 결과를 보였습니다"라고 **정량적**으로 말할 수 있게 됩니다. 이것이 **프롬프트 엔지니어링의 완성**입니다.

> 📷 **[이미지]** A/B 테스트 결과를 요약한 표 또는 대시보드. [테스트 케이스 ID] | [Prompt A 답변] | [Prompt B 답변] | [평가 점수] 컬럼으로 구성.

> **📌 참고:** LLM-as-a-Judge 기법을 사용하면 대규모 평가를 자동화할 수 있습니다.
