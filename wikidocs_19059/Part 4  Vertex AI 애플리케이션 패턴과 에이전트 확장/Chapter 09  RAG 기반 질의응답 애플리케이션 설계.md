우리는 지난 Chapter 8을 통해 방대한 문서 중에서 사용자의 질문과 가장 관련된 내용을 찾아내는 **검색 엔진(Retriever)**을 성공적으로 구축했습니다. 하지만 검색된 결과는 아직 기계적인 텍스트 조각들의 나열일 뿐입니다.

일반 사용자에게 "관련 문서는 doc_05, doc_08입니다"라고 던져주는 것은 친절한 서비스가 아닙니다.

우리가 원하는 것은 "검색된 문서를 바탕으로, 사용자의 질문에 대해 사람처럼 자연스럽게 설명해 주는 답변"입니다. 이를 위해 검색된 텍스트를 **LLM(Gemini)**에게 전달하고, "이 내용을 읽고 답변을 작성해 줘"라고 시키는 과정이 필요합니다.

이번 챕터에서는 **검색기(Retriever)**와 **생성기(Generator)**를 결합하여 완전한 **RAG(Retrieval-Augmented Generation)** 파이프라인을 구축하고, 답변의 신뢰도를 높이기 위해 **근거(출처)**를 함께 제시하는 방법을 상세히 알아보겠습니다.

> **💡 핵심 포인트:** 검색과 생성이 결합된 RAG 시스템은 사용자의 질문에 근거 기반의 자연스러운 답변을 제공합니다.

## 9-1 Retriever-Generator 구조 상세 분석

RAG 아키텍처는 크게 정보를 찾는 **검색기(Retriever)**와 정보를 바탕으로 글을 쓰는 **생성기(Generator)**라는 두 개의 두뇌로 이루어져 있습니다. 이 둘이 어떻게 협력하는지 상세 과정을 이해해야 좋은 애플리케이션을 설계할 수 있습니다.

### 9-1-1 오픈북 시험(Open-book Exam) 비유

RAG 시스템의 작동 원리는 학생이 치르는 '오픈북 시험'과 매우 흡사합니다.

- **학생 (Generator/LLM)**: 똑똑하고 글을 잘 쓰지만, 시험 범위의 내용을 전부 외우지는 못했습니다.
- **교과서 (Vector Store)**: 정답이 적혀 있는 방대한 지식 저장소입니다.
- **인덱스/목차 (Retriever)**: 교과서의 어느 페이지에 관련 내용이 있는지 빠르게 찾아주는 도구입니다.

시험 문제가 나오면(질문),

1. 학생은 먼저 목차를 뒤져서 관련 페이지를 펼칩니다(검색).
2. 그리고 그 페이지의 내용을 읽고 이해한 뒤, 자신의 문장력으로 답안지를 작성합니다(생성).

RAG도 정확히 이 순서대로 동작합니다.

### 9-1-2 데이터 흐름의 5단계

실제 시스템 내부에서는 다음과 같은 5단계의 데이터 흐름이 발생합니다.

1. **질문(Query)**: 사용자가 "RAG의 장점이 뭐야?"라고 묻습니다.
2. **검색(Retrieve)**: 질문을 벡터로 변환해 벡터 스토어에서 관련 문서 조각 3~5개를 가져옵니다.
3. **프롬프트 주입(Prompt Stuffing)**: 가져온 문서 조각들을 프롬프트라는 하나의 그릇에 담습니다. "다음 내용을 참고해서..."라는 지시어와 함께 묶습니다.
4. **생성(Generate)**: 완성된 거대한 프롬프트를 Gemini에게 보냅니다.
5. **답변(Answer)**: Gemini가 문맥을 파악하고 요약하여 최종 답변을 생성합니다.

이 과정에서 가장 중요한 것은 3번 단계인 **프롬프트 주입**입니다. 검색된 내용을 어떻게 포장해서 AI에게 주느냐에 따라 답변 품질이 천차만별로 달라지기 때문입니다.

> 📷 **[이미지]** [User] -> [Question] -> [Retriever] -> [Documents] -> [Prompt Template] -> [LLM(Gemini)] -> [Answer]로 이어지는 RAG 파이프라인의 전체 흐름도. Document가 Prompt Template 안으로 쏙들어가는(Stuffing) 모습을 시각적으로 강조.

## 9-2 검색 결과를 프롬프트에 삽입하는 템플릿 설계

Gemini에게 검색 결과를 건네줄 때는 정해진 규칙, 즉 **템플릿(Template)**이 필요합니다. 단순히 텍스트를 붙여넣는 것이 아니라, AI가 해야 할 역할을 명확히 정의해 주어야 합니다.

### 9-2-1 기본 프롬프트 템플릿 구조

가장 널리 사용되는 프롬프트 템플릿은 다음과 같은 구조를 가집니다.

```
[시스템 지침]
당신은 친절한 AI 어시스턴트입니다. 아래 제공된 [Context]만을 사용하여
질문에 답하세요.
만약 [Context]에 정답이 없다면, "정보가 부족하여 알 수 없습니다"라고
말하고 지어내지 마세요.

[Context (검색된 내용)]
{context}

[Question (사용자 질문)]
{question}

[Answer (답변)]
```

여기서 `{context}` 자리에는 **검색기(Retriever)**가 찾아온 문서 내용들이 자동으로 들어가고, `{question}` 자리에는 사용자의 질문이 들어갑니다.

### 9-2-2 환각(Hallucination) 방지 전략

RAG의 핵심 목표 중 하나는 AI가 거짓말을 하지 않게 하는 것입니다. 이를 위해 프롬프트에 강력한 제약 조건을 걸어야 합니다. 다음과 같은 조건을 포함해 보세요.

- "제공된 문맥(Context)에 있는 내용으로만 대답해라."
- "사전 지식을 사용하지 마라."
- "모르면 모른다고 해라."

이러한 지침이 포함되어야 Gemini가 자신의 훈련된 지식(인터넷 정보)과 검색된 기업 내부 정보(보안 문서)를 섞어서 엉뚱한 답을 내놓는 실수를 방지할 수 있습니다.

**LangChain**에서는 **PromptTemplate** 객체를 사용하여 이러한 구조를 코드로 쉽게 구현할 수 있습니다.

> 📷 **[이미지]** '나쁜 프롬프트'(단순히 질문만 던짐)와 '좋은 프롬프트'(역할, 제약조건, 문맥이 구조화됨)를 좌우로 비교. 좋은 프롬프트 쪽에는 각 섹션(지침, 문맥, 질문)이 박스로 구분되어 깔끔하게 정리된 모습.

## 9-3 근거(출처)를 포함하는 답변 포맷

기업용 검색 시스템에서 가장 중요한 것은 **신뢰성**입니다. AI가 그럴듯한 답변을 내놓더라도, 사용자는 "이거 진짜 맞는 말이야?"라고 의심할 수 있습니다. 이때 답변과 함께 **출처(Source)**를 표기해 준다면 신뢰도가 비약적으로 상승합니다.

### 9-3-1 메타데이터의 활용

우리는 Chapter 7에서 문서를 자를 때 **파일명(source)**과 **페이지 번호(page)**를 **메타데이터**로 저장해 두었습니다. 검색된 문서 조각(**Document 객체**)에는 이 정보가 그대로 살아 있습니다.

### 9-3-2 출처 표기 전략

답변을 생성할 때, 두 가지 방식으로 출처를 보여줄 수 있습니다.

**방식 A: 답변 내 인용 (Inline Citation)**

- AI에게 지시하여 문장 끝에 주석을 달게 하는 방식입니다.
- 예: "Vertex AI는 머신러닝 플랫폼입니다 [참고: guide.pdf, 3쪽]."
- 난이도가 높고 모델 성능에 따라 누락될 수 있습니다.

**방식 B: 답변 하단 부록 (Recommended)**

- LLM이 답변을 생성한 후, 프로그램(Python 코드) 레벨에서 검색에 사용된 문서의 메타데이터를 하단에 리스트로 붙여주는 방식입니다.
- 구현이 쉽고 확실하게 출처를 명시할 수 있어 가장 권장되는 방식입니다.

이어지는 실습을 통해 **방식 B**를 채택하여, 답변 하단에 "**참고 문서:**" 리스트를 출력하도록 구현할 것입니다.

> 📷 **[이미지]** 최종 답변 화면 예시. 상단에는 AI의 자연어 답변이 나오고, 하단에는 구분선과 함께 [참고 문서 목록] 1. vertex_guide.pdf (p.15) 2. api_spec.docx (p.3) 처럼 출처가 깔끔하게 나열된 UI 목업.

## 9-4 RAG 기반 Q&A 에이전트 구현

이제 이론적인 준비는 끝났습니다. Chapter 8에서 만든 **백엔드(RAGBackend)**와 **프롬프트 템플릿**, 그리고 **Gemini**를 결합하여 실제 작동하는 질의응답 에이전트 코드를 완성해 보겠습니다.

### 9-4-1 라이브러리 임포트 및 설정

먼저 **LangChain**의 핵심 모듈들을 불러옵니다.

```python
from langchain.prompts import PromptTemplate
from langchain.chains import RetrievalQA
from langchain_google_vertexai import VertexAI

# Chapter 8에서 만든 벡터 DB 클래스 임포트 (파일이 분리되어 있다고 가정)
# from rag_backend import RAGBackend
# 실습 편의상 여기서는 직접 vector_store를 로드한다고 가정합니다.
```

> **📌 참고:** 위 코드는 LangChain 라이브러리 기반의 패턴입니다.

### 9-4-2 프롬프트 템플릿 정의

앞서 설계한 **환각 방지** 프롬프트를 코드로 정의합니다.

```python
template = """
당신은 기업 내부 문서를 기반으로 답변하는 전문 어시스턴트입니다.
반드시 아래의 [Context]에 있는 내용만을 바탕으로 답변하세요.
만약 문맥에서 답을 찾을 수 없다면 "문서에 해당 내용이 없습니다"라고 답하세요.

[Context]
{context}

[Question]
{question}

[Answer]
"""

QA_CHAIN_PROMPT = PromptTemplate(
    input_variables=["context", "question"],
    template=template,
)
```

> **⚠️ 주의:** 템플릿의 `{context}`와 `{question}` 변수는 실행 시점에 자동으로 치환됩니다.

### 9-4-3 체인(Chain) 조립 및 실행 함수

이제 **LLM**, **검색기(Retriever)**, **프롬프트**를 연결하는 **체인**을 만듭니다. LangChain의 **RetrievalQA** 체인을 사용하면 이 복잡한 연결을 한 줄로 처리할 수 있습니다.

```python
# 1. LLM 설정 (Gemini Pro)
llm = VertexAI(model_name="gemini-1.5-pro-001", temperature=0)

# 2. 체인 생성
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    chain_type="stuff",  # 검색 결과를 통째로 프롬프트에 넣는 방식
    retriever=rag_backend.vector_store.as_retriever(),  # Ch8의 검색기
    return_source_documents=True,  # 출처 확인을 위해 원본 문서 반환 설정
    chain_type_kwargs={"prompt": QA_CHAIN_PROMPT}
)

def ask_question(query):
    # 체인 실행
    result = qa_chain.invoke({"query": query})

    # 결과 파싱
    answer = result["result"]
    sources = result["source_documents"]

    # 출력
    print(f"\n[질문]: {query}")
    print(f"[답변]: {answer}")
    print("-"*30)
    print("[참고 문서]")
    for doc in sources:
        source_name = doc.metadata.get('source', 'Unknown')
        page_num = doc.metadata.get('page', 0)
        print(f"- {source_name} (p.{page_num})")
```

> **📌 참고:** `return_source_documents=True` 옵션은 검색에 사용된 원본 문서를 반환하여 출처 표시를 가능하게 합니다.

### 9-4-4 최종 테스트

이제 질문을 던져보면, 에이전트는 벡터 DB를 뒤져서 관련 내용을 찾고, **Gemini**가 이를 정리하여 답변한 뒤, 우리가 의도한 대로 출처까지 함께 보여줄 것입니다.

```python
ask_question("RAG 파이프라인 구축 시 주의할 점은?")
```

**실행 결과 예시:**

```
[답변]: RAG 파이프라인 구축 시에는 데이터 전처리와 청킹 전략이 매우 중요합니다. 또한 검색된 문맥이 너무 길 경우 LLM의 토큰 제한을 넘을 수 있으므로 주의해야 합니다.

[참고 문서]
vertex_rag_guide.pdf (p.12)
vertex_rag_guide.pdf (p.13)
```
