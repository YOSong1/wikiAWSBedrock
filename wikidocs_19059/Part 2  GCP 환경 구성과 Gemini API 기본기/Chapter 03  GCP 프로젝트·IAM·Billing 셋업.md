생성형 AI 애플리케이션 개발은 코드를 짜는 것에서 시작하지 않습니다. 건물을 지을 때 지반을 다지고 울타리를 치듯, 클라우드 환경에서도 **작업 공간(Project)**을 정의하고, **접근 권한(IAM)**을 통제하며, **비용 관리 체계(Billing)**를 세우는 것이 선행되어야 합니다.

> **📌 참고:**
> 이 장에서는 구글 클라우드(GCP) 환경을 '프로덕션 레벨'을 고려하여 제대로 세팅하는 방법을 상세히 다룹니다.

---

## 3-1 GCP 프로젝트와 리소스 계층 구조 이해

GCP를 처음 시작할 때 가장 헷갈리는 부분이 바로 **'프로젝트'**와 그 상위 개념들입니다. 개인이 혼자 개발할 때와 회사가 팀 단위로 개발할 때, GCP를 바라보는 관점이 완전히 다르기 때문입니다. 이 차이를 명확히 이해해야 나중에 "권한이 왜 없지?" 또는 "결제 관리가 왜 안 되지?" 같은 문제를 피할 수 있습니다.

### 3-1-1 개인 개발자 관점

**"모든 것은 프로젝트(Project) 안에 있다"**

여러분이 GCP에서 가상 컴퓨터(VM)를 만들거나, Vertex AI로 모델을 돌리려면 무조건 **'프로젝트'**라는 그릇을 먼저 만들어야 합니다.

**프로젝트의 역할:** 하나의 독립적인 **'작업 공간'**입니다.

**프로젝트의 특징:**

- API를 켜고 끄는 스위치가 프로젝트에 달려 있습니다.
- 신용카드(결제)를 프로젝트에 연결합니다.
- 팀원을 초대(IAM)하는 공간도 프로젝트입니다.

개인 학습이나 소규모 개발 시에는 프로젝트 하나만 잘 만들면 됩니다. 상위 구조는 몰라도 전혀 지장이 없습니다.

---

### 3-1-2 기업/조직 관점

**"프로젝트를 담는 폴더 구조가 필요하다"**

기업 또는 조직의 관점에서 필요한 개념을 정리해 보겠습니다.

마케팅팀, 개발팀, 데이터팀이 각각 수십 개의 프로젝트를 만든다고 상상해 보세요. 수백 개의 프로젝트를 일일이 관리하는 것은 불가능합니다. 그래서 기업 환경에서는 프로젝트보다 그 상위의 **'계층 구조(Hierarchy)'** 설계가 훨씬 중요해집니다.

GCP는 이를 위해 윈도우 탐색기 폴더처럼 정리할 수 있는 구조를 제공합니다.

**조직 (Organization)** [가장 높은 회사 레벨]

회사 전체를 뜻합니다. 여기서 "모든 프로젝트는 2단계 인증 필수"라는 보안 규칙을 걸면, 회사 내 수천 개의 프로젝트에 강제로 적용됩니다. (가장 강력한 관리자)

**폴더 (Folder)** [부서나 팀 레벨]

'개발팀 폴더', '서비스팀 폴더' 등으로 프로젝트를 묶습니다.

'개발팀 폴더'에 팀장님에게 권한을 주면, 그 안에 있는 모든 프로젝트의 권한을 팀장님이 자동으로 갖게 됩니다. (권한 상속)

**프로젝트 (Project)** [실제 작업 공간]

앞서 설명한 실제 리소스가 담기는 곳입니다.

---

### 3-1-3 구글 클라우드의 리소스 계층 구조

GCP는 리소스를 관리하기 위해 계층적인 컨테이너 구조를 제공합니다. 이는 윈도우의 폴더 구조와 유사하지만, **정책 상속(Policy Inheritance)**이라는 강력한 특징을 가집니다.

**조직 (Organization)** [Root Node]

회사 도메인(예: samsung.com, startup.io)과 연결되는 최상위 컨테이너입니다. 여기서 설정한 보안 정책(예: "모든 프로젝트는 2단계 인증 필수")은 하위의 모든 리소스에 강제 적용됩니다.

**폴더 (Folders)**

조직 밑의 부서(팀) 또는 환경(Dev/Prod)을 나누는 단위입니다. IAM 권한을 폴더 단위로 부여하면, 해당 폴더 내의 모든 프로젝트에 권한이 자동 상속됩니다.

**프로젝트 (Projects)**

가장 핵심적인 관리 단위입니다. API 활성화, 결제 연결, 할당량(Quota) 제한이 모두 프로젝트 단위로 이루어집니다.

**리소스 (Resources)**

실제 서비스들입니다. (예: Vertex AI Endpoint, Cloud Storage Bucket, BigQuery Dataset)

---

### 3-1-4 프로젝트 생성과 ID 명명 규칙

프로젝트 이름은 나중에 바꿀 수 있지만, **프로젝트 ID는 생성 후 절대 변경할 수 없습니다.** 따라서 명명 규칙이 매우 중요합니다.

**콘솔 접속**

Google Cloud Console에 로그인합니다.

**새 프로젝트**

상단 프로젝트 선택기 > [새 프로젝트] 클릭.

**정보 입력**

- 프로젝트 이름: 식별하기 쉬운 이름 (예: AI Chatbot Lab)
- 프로젝트 ID (중요): 시스템이 임의로 부여하는 난수(예: ai-chatbot-34921)를 그대로 쓰지 말고, 의미 있는 ID로 수정하세요.

**권장 패턴**

`<회사/팀명>-<환경>-<서비스명>` (예: myteam-dev-genai-bot)

---

### 3-1-5 GCP 프로젝트 삭제

GCP(구글 클라우드) 프로젝트는 **"즉시 완전 삭제"**가 아니라 **Shut down(종료/삭제 요청)** 형태로 들어가고, 보통 30일 유예기간 후 영구 삭제됩니다.

#### 3-1-5-1 콘솔(UI)에서 프로젝트 삭제(Shut down)하기

- Google Cloud Console 접속 → 상단의 프로젝트 선택(지금 화면)에서 대상 프로젝트 확인
- 좌측 메뉴(≡) → IAM 및 관리자(IAM & Admin) → 설정(Settings) 로 이동
- 설정 화면에서 삭제할 프로젝트를 선택 → Shut down(종료) 클릭
- 확인 창에서 프로젝트 ID를 입력하고 Shut down anyway(종료)로 확정

> **⚠️ 주의:**
> Shut down 버튼이 안 보이면 해당 프로젝트에서 `resourcemanager.projects.delete` 권한이 없어서입니다.

#### 3-1-5-2 gcloud(Cloud Shell/로컬)로 삭제하기

```bash
gcloud projects delete PROJECT_ID
```

이 명령도 프로젝트를 즉시 "삭제 예약(삭제 표시)" 상태로 만들고, 동일하게 유예기간 후 영구 삭제됩니다.

#### 3-1-5-3 삭제 취소(복구) 가능 기간과 방법

프로젝트는 삭제 후 30일 안에 복구 가능하고, 이후에는 복구할 수 없습니다.

**gcloud로 복구**

```bash
gcloud projects undelete PROJECT_ID
```

#### 3-1-5-4 삭제 전에 꼭 알아둘 주의사항

> **⚠️ 주의:**
> - 프로젝트가 삭제 표시되면 바로 사용 불가가 되고, 30일 후 완전 삭제됩니다.
> - Billing 연결은 끊기며, 복구하더라도 자동으로 다시 연결되지 않습니다.
> - 일부 리소스(예: Cloud Storage, Pub/Sub 등)는 더 빨리 삭제되어 30일 내 복구하더라도 완전 복구가 안 될 수 있습니다.
> - 예기치 않은 과금 방지를 위해 프로젝트 Shut down 전에 Billing 비활성화를 권장합니다.
> - 삭제된 프로젝트 ID는 재사용 불가입니다. (나중에 같은 ID를 쓰고 싶다면 "프로젝트 삭제" 대신 "리소스만 정리"가 더 안전)

## 3-2 IAM 역할(Role)과 서비스 계정 보안 전략

**보안 사고의 90%는 잘못된 권한 설정에서 발생합니다.** **IAM(Identity and Access Management)**을 단순히 '로그인 기능'이 아니라, **'최소 권한 원칙(Principle of Least Privilege)'**을 구현하는 도구로 이해해야 합니다.

---

### 3-2-1 IAM 역할의 3가지 유형

단순히 "관리자 권한 주세요"라고 하면 보안 구멍이 생깁니다. 구글은 역할을 세분화하여 제공합니다.

**기본 역할 (Primitive Roles)**

Owner, Editor, Viewer. 과거의 유산으로, 권한이 너무 광범위합니다. 실무에서는 절대 사용을 지양해야 합니다. (예: Editor 권한을 주면 AI 모델만 수정하는 게 아니라, 네트워크 설정까지 바꿔버릴 수 있습니다.)

**사전 정의된 역할 (Predefined Roles)**

구글이 서비스별로 쪼개 놓은 역할입니다.

- Vertex AI 관리자 (roles/aiplatform.admin): 모델 생성, 배포, 삭제 가능.
- Vertex AI 사용자 (roles/aiplatform.user): 모델 예측 및 리소스 조회만 가능.

**커스텀 역할 (Custom Roles)**

특정 API 하나만 호출할 수 있게 아주 잘게 쪼갠 역할입니다.

---

### 3-2-2 팀원에게 Vertex AI 개발 권한 부여하기

- [IAM 및 관리자] > [IAM] 메뉴 진입.
- [액세스 권한 부여] 클릭.
- 새 주체: 팀원의 구글 이메일 입력.
- 역할 선택: 필터에 'Vertex AI' 검색 후 Vertex AI User 선택. (관리자가 아니라면 User 권한으로 충분합니다.)

![IAM 역할 선택 드롭다운 메뉴](./images/chapter3_1.png)

> **📌 참고:**
> 역할 선택 시 'Editor' 같은 광범위한 권한 대신, Vertex AI User처럼 구체적인 서비스별 역할을 선택하는 것이 보안의 핵심입니다.

## 3-3 Billing 연결과 비용 폭탄 방지책

**AI 서비스, 특히 LLM은 토큰 단위로 과금되므로 비용 예측이 어렵습니다.** 단순한 '알림' 설정을 넘어, 데이터를 분석할 수 있는 체계를 갖춰야 합니다.

---

### 3-3-1 결제 데이터 내보내기 (Billing Export)

단순히 이번 달 얼마가 나왔는지 보는 것을 넘어, "어떤 쿼리가 비용을 많이 유발했는지" 분석하려면 **Billing Export** 설정이 필수입니다.

**기능**

매일 발생하는 상세 과금 로그를 BigQuery 데이터셋으로 자동 적재합니다.

**활용**

SQL을 이용해 "지난주 Gemini Pro 모델 호출 비용만 시간대별로 추출"하거나 "가장 비싼 쿼리를 날린 사용자"를 찾아낼 수 있습니다.

**설정 방법**

[결제] > [결제 내보내기] > [BigQuery 내보내기] 편집 > 프로젝트 및 데이터셋 지정.

---

### 3-3-2 다중 안전장치: 예산 알림 설정

알림은 한 번만 받으면 놓치기 쉽습니다. 단계별로 설정하세요.

**메뉴 경로**

[결제] > [예산 및 알림] > [예산 만들기]

**범위**

프로젝트 전체 선택.

**알림 기준 (Thresholds)**

- 50% 도달 시: "주의" (이메일 발송)
- 90% 도달 시: "경고" (이메일 + 슬랙 연동 권장)
- 100% 도달 시: "비상"

## 3-4 Vertex AI 리전 전략과 할당량(Quota) 확인

Vertex AI를 사용할 때 **"어느 리전(Region)을 쓸 것인가"**는 속도와 기능, 두 마리 토끼를 잡는 결정입니다.

---

### 3-4-1 리전 선택 가이드: us-central1 vs asia-northeast3

**us-central1 (아이오와)**

구글의 본진입니다. Gemini 2.5 Pro, Imagen 3 등 최신 모델과 기능이 가장 먼저 출시됩니다. GPU 자원도 가장 풍부합니다. 초기 개발이나 최신 기능 테스트 시 강력 추천합니다.

**asia-northeast3 (서울)**

한국 사용자 대상의 서비스라면 네트워크 지연 시간(Latency)이 가장 적습니다. 데이터가 국내에 저장되어야 하는 컴플라이언스 이슈가 있다면 필수입니다.

**결론**

개발 및 PoC 단계에서는 us-central1을 사용하고, 실제 대고객 서비스 런칭 시점에 asia-northeast3의 모델 지원 여부를 확인 후 전환하는 전략이 유효합니다.

---

### 3-4-2 API 활성화 및 할당량 확인

**API 활성화**

콘솔 상단 검색창에 Vertex AI API 검색 > [사용] 클릭.

**할당량 확인**

- [IAM 및 관리자] > [할당량 및 시스템 한도] 메뉴 이동.
- 필터에 Generative AI API 검색.
- Requests per minute (분당 요청 수) 한도를 확인합니다. 기본적으로 분당 60회 정도로 제한되어 있을 수 있으며, 상용화 전에는 반드시 **[할당량 수정]**을 눌러 증설 요청을 해야 합니다.
