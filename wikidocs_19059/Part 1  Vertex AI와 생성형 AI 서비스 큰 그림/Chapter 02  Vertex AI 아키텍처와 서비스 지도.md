## 2-1 Vertex AI 주요 컴포넌트 개요

이번 챕터에서는 Google Cloud의 AI 통합 플랫폼인 **Vertex AI**를 구성하는 핵심 '**빌딩 블록(Building Blocks)**'들을 살펴봅니다.

**Vertex AI**는 단순히 모델을 제공하는 곳이 아니라, 모델의 발견(**Discovery**) → 튜닝(**Tuning**) → 개발(**Development**) → 배포(**Deployment**) → 관리(**Management**)에 이르는 전 과정을 지원하는 **엔드투엔드(End-to-End) 플랫폼**입니다.

우리가 앱을 개발할 때 거치게 될 작업 흐름 순서대로 각 컴포넌트를 상세히 알아보겠습니다.

> 🔗 **[콘솔 접속]** https://console.cloud.google.com/

### 2-1-1 Model Garden (모델 가든): AI 모델 탐색 및 선택

가장 먼저 방문하게 될 곳은 **Model Garden**입니다. 이곳은 일종의 "**AI 모델 앱스토어**" 역할을 합니다.

[그림 2-3] Vertex AI Model Garden 인터페이스와 모델 분류![](https://static.wikidocs.net/images/page/327186/chapter2_3.png)

#### 역할 및 기능

- **멀티 벤더 허브**: Google의 자체 모델(Gemini, PaLM, Imagen)뿐만 아니라, 메타(Meta)의 Llama 3, 미스트랄(Mistral), 앤스로픽(Anthropic)의 Claude 3 등 전 세계적으로 인기 있는 모델들을 한곳에 모아두었습니다.

- **원클릭 배포 (One-Click Deploy)**: 오픈소스 모델을 직접 서버에 설치하려면 복잡한 환경 설정이 필요하지만, 모델 가든에서는 'Deploy' 버튼 하나로 내 프로젝트의 엔드포인트(VM)에 모델을 올릴 수 있습니다.

- **모델 카드 (Model Card)**: 각 모델을 클릭하면 모델의 성능표(Benchmark), 사용 비용, 입력 토큰 한도, 저작권 라이선스 등 상세 정보를 명세서처럼 확인할 수 있습니다.

#### 활용 시나리오

- 프로젝트 기획 초기 단계에서 "우리 서비스에 적합한 모델은 무엇인가?"를 비교 분석할 때 사용합니다.

- **Gemini API** 키를 발급받거나, **Llama** 모델을 내 전용 서버에 띄우고 싶을 때의 시작점입니다.

### 2-1-2 Generative AI Studio (Vertex AI Studio): 노코드 실험 및 튜닝

모델을 골랐다면, 코드를 짜기 전에 그 모델이 내 의도대로 답하는지 테스트해야 합니다. 이곳이 바로 **Vertex AI Studio**입니다.

[그림 2-4] Generative AI Studio (Vertex AI Studio) 콘솔 메뉴![](https://static.wikidocs.net/images/page/327186/chapter2_4.png)

인터페이스 구성:
- 좌측: 채팅창 영역
- 우측: 설정 패널 (Temperature, Token limit, Safety settings)
- 상단: 'View Code' 버튼

#### 역할 및 기능

- **프롬프트 디자인 (Prompt Design)**: 챗GPT와 유사한 채팅 인터페이스를 제공합니다. 여기서 개발자와 기획자는 '**시스템 지침(System Instruction)**'을 입력하여 모델에게 페르소나(예: "너는 친절한 여행 가이드야")를 부여하고 테스트할 수 있습니다.

- **파라미터 실험실**:
  - **Temperature (온도)**: 모델의 창의성을 조절합니다. (0에 가까우면 사실적/보수적, 1에 가까우면 창의적/랜덤성 증가)
  - **Output Token Limit**: 답변의 길이를 제한합니다.
  - **Safety Filter**: 유해성(혐오 발언, 성적 콘텐츠 등) 차단 강도를 설정합니다.

- **파인 튜닝 (Fine-tuning) UI**: 복잡한 파이썬 코드 없이, 준비된 데이터셋(JSONL 파일)을 업로드하고 버튼을 누르는 것만으로 모델을 미세조정(Tuning)하는 파이프라인을 실행할 수 있습니다.

#### 핵심 차별점

- **View Code**: 스튜디오에서 테스트를 마친 프롬프트 설정값은 우측 상단 'Get Code' 버튼을 눌러 즉시 **Python**, **Node.js**, **cURL** 코드로 변환하여 복사할 수 있습니다. 이는 개발 시간을 획기적으로 단축해 줍니다.

### 2-1-3 Vertex AI Workbench: 통합 개발 환경

프롬프트 검증이 끝났다면, 실제 애플리케이션 로직을 개발해야 합니다.

**Workbench**는 데이터 과학자와 AI 엔지니어를 위한 **클라우드 기반 통합 개발 환경(IDE)**입니다.

#### 역할 및 기능

- **Managed Notebooks**: 로컬 컴퓨터에 Python, CUDA(GPU 드라이버), TensorFlow/PyTorch 등을 설치하느라 고생할 필요가 없습니다. Google이 미리 세팅해둔 보안 환경의 인스턴스를 몇 분 만에 생성하여 접속합니다.

- **하드웨어 유연성**: 코딩하다가 메모리가 부족하거나 GPU가 필요하면, 인스턴스 설정을 변경하여 스펙을 즉시 업그레이드할 수 있습니다.

- **데이터 통합**: Google Cloud의 데이터 웨어하우스인 **BigQuery**, 스토리지인 **Cloud Storage**와 네트워크적으로 긴밀히 연결되어 있어, 대용량 데이터를 빠르고 안전하게 로드할 수 있습니다.

#### 활용 시나리오

- **RAG(검색 증강 생성) 시스템 구축**을 위한 데이터 전처리 코드 작성.

- **Vertex AI SDK**를 활용한 챗봇 백엔드 로직 구현 및 테스트.

### 2-1-4 Endpoints: 모델 배포 및 서빙

우리가 만든 모델이나 튜닝된 모델을 실제 서비스(웹/앱)에서 호출하려면 API 주소가 필요합니다. **Endpoints**는 모델을 온라인 서비스로 만들어주는 접점입니다.

#### 역할 및 기능

- **실시간 예측 (Online Prediction)**: 사용자가 질문하면 즉시 답변해야 하는 챗봇 서비스 등을 위해, HTTP REST API 형태로 모델을 항상 켜두는 방식입니다.

- **오토 스케일링 (Auto-scaling)**: 사용자가 몰릴 때는 자동으로 서버(노드) 수를 늘리고, 사용자가 없을 때는 줄여서 비용을 최적화합니다.

- **트래픽 분할 (Traffic Splitting)**: 신규 모델을 배포할 때, 전체 사용자의 10%만 새 모델로 보내고 90%는 기존 모델로 보내는 식의 **A/B 테스트**를 설정만으로 수행할 수 있습니다.

> **💡 핵심 포인트:** Gemini와 같은 Google의 파운데이션 모델은 이미 Google이 관리하는 공용 엔드포인트가 있으므로, 사용자가 별도로 Endpoints를 생성할 필요 없이 API 키만 있으면 호출 가능합니다.

### 2-1-5 Vertex AI Pipelines: MLOps와 자동화

일회성 개발이 아니라, 지속적으로 데이터가 들어오고 모델이 업데이트되어야 한다면 자동화가 필요합니다. **Pipelines**는 머신러닝 워크플로를 자동화하는 도구입니다.

#### 역할 및 기능

- **워크플로 오케스트레이션**: "데이터 전처리 → 학습 → 평가 → 배포"라는 일련의 과정을 하나의 파이프라인으로 정의합니다.

- **서버리스 실행**: 파이프라인의 각 단계(Step)가 실행될 때만 컴퓨팅 자원을 쓰고, 끝나면 반납하므로 효율적입니다.

- **재현성(Reproducibility) 보장**: 언제 누가 돌려도 동일한 과정을 거치도록 보장하며, 과거의 실행 이력을 모두 추적할 수 있습니다.

#### 활용 시나리오

- 매주 새로운 문서가 업데이트될 때마다 자동으로 벡터 임베딩을 다시 생성하고 검색 인덱스를 갱신하는 **RAG 파이프라인** 구축 시 사용합니다.

## 2-2 Vertex AI 핵심 서비스 관계 및 활용

---

### 2-2-1 Vertex AI 핵심 서비스 간의 상호 작용 개요

Vertex AI에서 제공하는 다양한 생성형 AI 서비스는 단순히 독립적인 모델이 아니라, 상호 보완적인 관계를 맺으며 복잡하고 지능적인 애플리케이션을 구현하는 데 사용됩니다.

이 관계를 이해하는 핵심은 서비스를 다음 두 가지 축으로 나누어 보는 것입니다:

- **기반 모델의 능력 확장**: Text → Chat → Multimodal로 발전하며 모델이 처리할 수 있는 데이터의 종류와 대화의 복잡성이 증가합니다.

- **외부 지식의 연결**: **Embeddings**는 기반 기술로 작용하며, **Search (RAG)** 패턴을 구동하여 모델의 지식을 최신 정보나 사내 데이터로 확장합니다.

---

### 2-2-2 기반 모델: 능력의 확장 단계 (Text, Chat, Multimodal)

#### (1) Text (LLMs)와 Chat의 관계

**Text 모델**은 가장 기본적인 **거대 언어 모델(LLM)**의 형태를 대표하며, 주로 **단일 작업(Single-turn)**에 초점을 맞춥니다. 예를 들어, 문서를 요약하거나, 짧은 질문에 답변하거나, 문장을 번역하는 등 한 번의 입력과 한 번의 출력이 이루어지는 형태입니다.

반면, **Chat 모델**은 Text 모델의 기본적인 능력 위에 '**대화 기록(Context) 관리**' 메커니즘이 추가된 활용 패턴입니다. 사용자와 모델 간의 자연스러운 **다중 턴(Multi-turn) 대화**가 가능하도록 이전의 발언 내용을 기억하고 이를 바탕으로 일관된 답변을 생성합니다. 따라서 Chat은 Text 모델을 장기간의 상호작용에 최적화하여 활용하는 방식이라 할 수 있습니다.

**Text 모델 예시**:
Vertex AI의 Generative AI Studio에서 **gemini-2.5-flash** 모델을 사용하여 '다음 문단을 5줄로 요약해줘'와 같이 명확히 정의된 단일 작업을 요청하는 경우입니다.

**Chat 모델 예시**:
동일한 모델을 Chat 모드에서 사용하며, '이전에 말했던 내용을 바탕으로 다시 설명해줘'와 같이 연속적인 대화 기록을 참조해야 하는 경우입니다.

#### (2) Multimodal: 차원을 넘어서는 확장

**Multimodal 모델 (다중 모달 모델)**은 Text 및 Chat 모델의 능력을 비정형 데이터 영역으로 확장합니다. 기존 모델이 텍스트만 이해했다면, **Multimodal 모델**은 텍스트 외에 **이미지**, **오디오**, **비디오** 등의 다양한 입력 형태를 동시에 이해하고 처리할 수 있습니다.

**Multimodal 모델**은 Chat의 자연스러운 대화 능력과 결합하여, 사용자가 "이 이미지에 있는 제품의 특징을 설명해줘"라고 요청했을 때 이미지를 보고 텍스트로 자연스럽게 대화하는 시나리오를 가능하게 합니다. 이는 **기반 모델의 능력 중 가장 포괄적이고 진보된 형태**입니다.

**Multimodal 모델 예시**:
**Gemini 2.5 Pro**와 같은 모델을 사용합니다. 예를 들어, 스캔한 차트 이미지와 차트에 대한 질문 텍스트를 동시에 입력하면, 모델은 이미지를 분석하여 데이터 트렌드를 읽고 텍스트로 결과를 설명해줍니다. 이는 기존의 Text 모델로는 불가능했던 작업입니다.

### 2-2-3 지식 연결: Embeddings와 Search (RAG)의 역할

생성형 AI 모델의 근본적인 한계는 **학습 시점 이후의 최신 정보**나 **기업 내부의 비공개 데이터**에 접근할 수 없다는 점입니다. **Embeddings**와 **Search (RAG)**는 이 한계를 극복하기 위한 핵심적인 기술과 활용 패턴입니다.

#### (1) Embeddings: 벡터 공간에서의 의미 부여

**Embeddings**는 텍스트, 이미지 등 모든 형태의 데이터를 **고차원 공간의 숫자 배열(벡터)**로 변환하는 기술입니다. 이 벡터 공간에서는 의미적으로 유사한 데이터일수록 서로 가까운 거리에 배치됩니다.

**핵심 역할**: 인간의 언어와 지식을 기계가 계산할 수 있는 수학적 유사성으로 변환하여, **단순한 키워드 검색**을 넘어 **의미 기반의 유사도 검색(Semantic Search)**을 가능하게 하는 근간 기술입니다.

**Embeddings 구현**: Vertex AI에서 **text-embedding-004**와 같은 Embeddings 모델을 API로 호출하여 문서나 텍스트를 벡터화합니다.

**벡터 저장소**: 이 벡터들은 **Vertex AI Vector Search (구 Vertex AI Matching Engine)**에 저장되어, 빠르고 대규모의 유사성 검색이 가능하도록 합니다.

#### (2) Search (RAG): 외부 지식 기반 생성 패턴

**Search**는 일반적으로 **검색 증강 생성(Retrieval Augmented Generation, RAG)** 아키텍처를 통해 구현됩니다. **RAG**는 Embeddings 기술을 활용하여 LLM의 답변을 외부의 신뢰할 수 있는 지식에 근거(Grounded)하도록 만듭니다.

##### RAG의 4단계 프로세스

**1단계. 색인 (Indexing)**
기업의 문서, 데이터베이스 등 외부 지식을 청크 단위로 나누고, 이를 Embeddings 모델을 사용하여 벡터로 변환하여 벡터 데이터베이스에 저장합니다.

**2단계. 검색 (Retrieval)**
사용자 질문이 들어오면, 이 질문 역시 Embeddings 모델로 벡터 변환됩니다. 이 질문 벡터와 저장된 지식 벡터 간의 유사도를 계산하여 가장 관련성이 높은 지식(소스 문서 청크)을 검색합니다.

**3단계. 프롬프트 구성 (Prompting)**
검색된 지식(Retrieval)은 답변 생성을 위한 컨텍스트로 사용되며, 사용자 질문과 함께 Text/Chat/Multimodal 모델에게 프롬프트 형태로 전달됩니다.

**4단계. 생성 (Generation)**
모델은 외부 지식이라는 명확한 근거를 바탕으로 사실에 기반한 정확하고 충실한 답변을 생성합니다.

#### RAG 통합 솔루션

- **통합 서비스**: GCP는 RAG를 쉽게 구현할 수 있도록 **Vertex AI Search and Conversation**이라는 통합 서비스를 제공합니다. 이 서비스는 데이터 저장(Cloud Storage), 벡터화(Embeddings), 검색(Vector Search), 그리고 답변 생성(LLM)까지의 모든 RAG 단계를 단일 UI와 API로 처리합니다.

- **환각(Hallucination) 방지**: RAG를 통해 모델이 근거 없는 정보를 생성하는 환각 현상을 최소화하고, 답변의 출처(**Source Citation**)를 제공하여 신뢰도를 높일 수 있습니다.

#### Vertex AI 서비스의 역할 분담 및 조합

Vertex AI 서비스는 다음과 같이 역할을 분담하며, 애플리케이션의 복잡성을 해결하기 위해 조합됩니다.

| 구분 | 컴포넌트 | 역할 및 관계 | 예시 (LLM과의 조합) |
|------|---------|-----------|------------------|
| **단일 턴 기반 모델** | **Text (LLMs)** | 텍스트 생성 및 처리의 기본 능력 제공 | **Text + Search**: 사내 문서를 요약하거나 분류. |
| **다중 턴 활용 패턴** | **Chat** | 컨텍스트 관리 기능을 더하여 다중 턴 대화를 가능하게 함. | **Chat + Search**: 대화 흐름을 유지하는 최신 정보 Q&A 봇. |
| **멀티 모달 기반 모델** | **Multimodal** | 텍스트, 이미지, 음성 등 여러 모달리티를 동시에 이해하고 처리. | **Multimodal + Search**: 이미지와 텍스트 기반 질문 모두에 답변. |
| **기반 기술** | **Embeddings** | 모든 데이터를 벡터로 변환하여 의미 기반의 검색을 가능하게 하는 핵심 기술. | Search (RAG) 구현의 필수 요소. |
| **활용 패턴** | **Search (RAG)** | Embeddings를 활용하여 모델의 지식을 외부 데이터로 확장하고, 환각(Hallucination)을 줄이는 아키텍처 패턴. | Text/Chat/Multimodal 모델의 지식 확장 엔진 역할. |

## 2-3 애플리케이션 관점에서 본 Vertex AI 서비스 조합 패턴

---

### 2-3-1 조합 패턴의 정의 및 중요성

앞서 살펴본 핵심 서비스들은 실제 애플리케이션 개발 시 단독으로 사용되기보다는, **비즈니스 목표** 달성을 위해 특정 목적을 가지고 **통합적으로 조합**됩니다.

이 '**조합 패턴**'은 곧 **애플리케이션의 아키텍처**를 결정하며, 필요한 기능과 성능을 구현하는 **핵심 설계 방식**이 됩니다.

---

### 2-3-2 핵심 애플리케이션 유형별 서비스 조합 패턴

#### 패턴 1: RAG 기반 지식 확장형 애플리케이션

**목표**: 모델의 범용 지식 외에, 최신 또는 사내 지식을 활용하여 정확하고 근거 있는 답변을 제공하는 것.

| 핵심 서비스 | 기능 및 역할 | GCP 컴포넌트 |
|-----------|-----------|-----------|
| **Search (RAG)** | 외부 지식(문서)을 벡터화하고 검색하는 핵심 엔진 | Vertex AI Search and Conversation 또는 Vector Search + Pipelines |
| **Chat/Text/Multimodal** | 검색된 지식을 바탕으로 최종 답변을 생성하고, 대화의 연속성을 유지 | Gemini Pro/Flash |
| **Multimodal (선택)** | 문서 내 이미지/도표 분석이 필요한 경우에 추가로 활용 | Gemini 2.5 Pro |

##### 조합 패턴 예시: 고객 서비스 봇이 내부 정책 매뉴얼을 기반으로 답변하는 경우

**①** 매뉴얼 (데이터) → Embeddings → Vector Search에 저장

**②** 고객 질문 → Embeddings → Vector Search에서 관련 정책 검색(Search)

**③** 검색 결과 + 질문 → Chat 모델로 전달 → 최종 답변 생성

---

#### 패턴 2: 다중 채널 인터페이스형 애플리케이션 (통합 고객 경험)

**목표**: 텍스트, 음성, 이미지 등 다양한 형태로 들어오는 사용자 요청을 하나의 통합된 모델로 처리하여 일관된 경험을 제공하는 것.

| 핵심 서비스 | 기능 및 역할 | GCP 컴포넌트 |
|-----------|-----------|-----------|
| **Multimodal** | 텍스트 외의 비정형 데이터를 이해하고 처리하는 통합 인터페이스 역할 | Gemini 2.5 Pro |
| **Chat** | 요청이 들어온 채널(앱, 웹)에 관계없이 대화 기록을 관리하고 연속적인 흐름을 제공 | Vertex AI Chat 또는 커스텀 API 통합 |
| **Embeddings/Search (선택)** | Multimodal 모델이 최신 지식이나 데이터베이스 내용을 참조해야 할 때 결합 | Vector Search |

##### 조합 패턴 예시: 스마트 카메라 시스템이 현장 상황을 분석하는 경우

**①** 카메라 영상 (데이터) → Multimodal 모델이 이미지 및 상황 분석

**②** 분석 결과 (텍스트) → Text 모델이 보고서 생성 또는 Chat으로 담당자에게 상황 전달

---

#### 패턴 3: 대규모 데이터 분석 및 처리형 애플리케이션 (분석 도우미)

**목표**: 정형 데이터(DB, 스프레드시트)나 대량의 비정형 데이터(로그, 문서)를 AI가 분석하고, 사용자에게 자연어 기반으로 통찰력을 제공하는 것.

| 핵심 서비스 | 기능 및 역할 | GCP 컴포넌트 |
|-----------|-----------|-----------|
| **Text (Code Generation)** | 사용자의 자연어 질문을 SQL, Python 코드 등으로 변환하여 데이터베이스를 쿼리 | Codey (Code-bison) |
| **Pipelines** | 데이터 준비, 모델 학습/튜닝, 추론 프로세스 전체를 자동화하고 관리 | Vertex AI Pipelines |
| **Embeddings/Search** | 분석 대상 비정형 데이터(문서 등)를 색인화하여 통찰력 추출에 활용 | Vector Search |

##### 조합 패턴 예시: 데이터 분석가가 자연어로 비즈니스 질문을 던지는 경우

**①** 사용자 질문 ("지난 분기 매출 상위 5개 지역은?") → Text (Codey)가 BigQuery SQL 코드로 변환

**②** SQL 실행 (BigQuery) → 결과 획득

**③** SQL 결과 → Text 모델이 자연어로 다시 설명하여 보고서 형태로 생성

---

### 2-3-3 Vertex AI 애플리케이션 설계 시 고려 사항

| 고려 사항 | 설명 | 조합 패턴에 미치는 영향 |
|---------|------|-----------------|
| **데이터 타입** | 다루는 데이터가 텍스트, 이미지, 또는 혼합형인지 여부 | 이미지/오디오가 많으면 **Multimodal 모델** 필수 |
| **대화 연속성** | 단순 질문/응답인지, 아니면 대화 흐름 유지가 필요한지 여부 | 대화 흐름이 중요하면 **Chat 모델** 필수 |
| **지식 근거** | 모델이 외부 데이터를 참조해야 하는지, 자체 학습 지식만으로 충분한지 여부 | 최신/사내 데이터가 필요하면 **Search (RAG) 패턴** 필수 |

## 2-4 예제로 보는 Vertex AI 기반 앱 구조

### 2-4-1 앱 구조의 구성 요소

Vertex AI 기반 애플리케이션 구조는 일반적으로 다음의 **세 가지 계층**으로 구성됩니다:

**①** **프론트엔드/인터페이스 (Front-end/Interface)**
사용자 요청을 받고 결과를 보여주는 영역 (웹, 모바일 앱, 채팅 봇 인터페이스 등).

**②** **백엔드/오케스트레이션 (Back-end/Orchestration)**
사용자 요청을 분석하고, 어떤 AI 서비스와 외부 시스템을 호출할지 결정하는 핵심 로직 영역. (Vertex AI Pipelines, Cloud Functions, GKE 등에서 구동)

**③** **AI 서비스 및 데이터 (AI Services & Data)**
실제 지능을 담당하는 LLM과 데이터 저장소. (Vertex AI Models, BigQuery, Vector Search 등)

### 2-4-3 통합된 Vertex AI 플랫폼의 가치

Vertex AI는 단순히 개별 모델을 제공하는 것을 넘어, **Pipelines**, **Vector Search**, **Search and Conversation** 등의 **엔터프라이즈급 기능**을 통해 복잡한 AI 앱을 **안정적이고 확장 가능**하게 구축할 수 있는 **통합 플랫폼**을 제공합니다.

애플리케이션 구조 설계 시 이 **세 가지 유형의 패턴**을 활용하면, **비즈니스 요구사항**에 맞는 **최적의 아키텍처**를 빠르게 도출할 수 있습니다.
